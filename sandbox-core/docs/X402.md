# x402 Payment Support — Design Sketch

## The Chain Split Problem

The core tension: does it make sense to charge x402 on Arbitrum mainnet but execute game moves off-chain (SQLite/Postgres) or on a different chain (Orbit)?

**Short answer:** x402 on mainnet + off-chain DB game = clean, fine model. The chain split only becomes a real problem when the game itself moves on-chain. At that point, x402 doesn't fit per-action gating.

### Option A — x402 as API access layer (current, off-chain game)

x402 payments gate the HTTP endpoints. You receive mainnet USDC; game state lives in Postgres/SQLite. tUSDC stays simulated. This is the only sensible path right now because:

- The game is already a FastAPI server with DB-backed state
- x402 is just a paywall on top of existing auth middleware
- No chain split problem because no on-chain game state exists to conflict with

The relationship is clean: **pay on-chain → play off-chain simulation**. Same as paying a SaaS with crypto.

### Option B — x402 + Orbit chain game state (future)

When the game moves fully on-chain to the Orbit chain, x402 on Arbitrum mainnet to pay for Orbit calls doesn't make sense — you'd be bridging real value to execute on a chain with no independent economic weight. The correct model there is:

- Orbit chain uses native USDC bridged from Arbitrum (real economic weight)
- Game actions are smart contract calls on Orbit (buy tokens, originate mortgage, etc.)
- x402 doesn't apply — players send txs directly to contracts and pay gas

x402 would still make sense as a **hosted relayer**: player pays x402 → you submit the tx and cover gas. Gasless UX with on-chain settlement.

## What to Charge For

Not every endpoint should be paywalled — that breaks bot integrations and creates bad UX. Reads should always be free.

| Endpoint | Cost (USDC) | Rationale |
|---|---|---|
| `POST /games` | $0.05 | One-time per game; funds the session |
| `POST /games/{id}/advance-turn` | $0.01 | Core game loop, per turn |
| `POST /games/{id}/trade` | $0.005 | Per transaction |
| `POST /games/{id}/mortgage` | $0.01 | Higher value action |
| `POST /games/{id}/refi` | $0.005 | |
| `POST /games/{id}/heloc/draw` | $0.005 | |
| `POST /games/{id}/heloc/repay` | $0.002 | |
| `POST /games/{id}/bots` | $0.005 | |
| `GET *` (all reads) | Free | Portfolio, feed, leaderboard, debt — never paywalled |
| Bot players (`is_bot=True`) | Exempt | You're paying for bot compute already |
| Admin routes (`ADMIN_API_KEY`) | Exempt | Same exemption as rate limits |
| Internal `sandbox_runner.py` calls | Exempt | Calls service layer directly, never hits HTTP middleware |

## Implementation

### Middleware placement

New `X402PaymentMiddleware` sits between `CORSMiddleware` and `ClerkAuthMiddleware`:

```
CORSMiddleware
  └─ X402PaymentMiddleware          ← NEW
       └─ ClerkAuthMiddleware
            └─ APIKeyMiddleware
                 └─ RateLimitMiddleware
                      └─ Route handler
```

Why middleware and not route-level decorators:
- Payment logic stays completely decoupled from game logic
- Works uniformly for Clerk-authed humans and `sb_`-key agents
- Price table is configurable externally without touching route code

### Request flow

```
Incoming POST /api/sandbox/games/{id}/trade
  │
  ├─ GET request? → pass through immediately
  ├─ Admin key or exempt route? → pass through
  ├─ Check PRICE_TABLE[route_pattern] → e.g. $0.005
  │
  ├─ X-PAYMENT header present?
  │    ├─ Verify via facilitator (or self-verify)
  │    ├─ Check payment_id not already used (dedup)
  │    ├─ Valid → attach payment_id to request state → pass through
  │    └─ Invalid → 402 with updated WWW-Authenticate
  │
  └─ No payment header → return 402 Payment Required
       body: {
         x402Version,
         accepts: [{
           scheme: "exact",
           network: "base-mainnet",
           maxAmountRequired: "5000",   // in USDC atomic units
           asset: USDC_BASE_ADDRESS,
           payTo: X402_PAYMENT_ADDRESS
         }]
       }
```

### New config vars (`app/core/config.py`)

```python
X402_ENABLED: bool = False           # feature flag — off by default
X402_PAYMENT_ADDRESS: str = ""       # USDC receive address on Base or Arbitrum
X402_NETWORK: str = "base-mainnet"   # base-mainnet recommended (lowest fees)
X402_FACILITATOR_URL: str = ""       # leave empty to self-verify
X402_PRICE_TABLE: dict = {
    "POST /api/sandbox/games$": 0.05,
    "POST /api/sandbox/games/.*/advance-turn": 0.01,
    "POST /api/sandbox/games/.*/trade": 0.005,
    "POST /api/sandbox/games/.*/mortgage": 0.01,
    "POST /api/sandbox/games/.*/refi": 0.005,
    "POST /api/sandbox/games/.*/heloc/draw": 0.005,
    "POST /api/sandbox/games/.*/heloc/repay": 0.002,
    "POST /api/sandbox/games/.*/bots": 0.005,
}
```

### New file: `app/core/x402_middleware.py`

Responsibilities:
1. Route pattern matching against `X402_PRICE_TABLE`
2. Parse and verify `X-PAYMENT` header (self-verify or forward to facilitator)
3. Payment deduplication — check `payment_id` against `SandboxTransaction.rentline_payment_id`
4. Attach `payment_id` to `request.state` for downstream use
5. Return well-formed 402 response when payment is absent or invalid

### State and idempotency

The real state concern with x402 is: what if payment goes through but the game move fails, or the response is dropped after payment?

x402 handles this at the protocol level via `X-Payment-Response` in the success response. Additional safeguards:

1. **Payment deduplication**: `SandboxTransaction.rentline_payment_id` already exists (null today). Store the `payment_id` / tx hash here. Reject duplicate `payment_id` before executing any state change.

2. **Atomic failure risk**: middleware verifies payment, then route handler executes. If the handler throws, payment is already taken. Options:
   - Accept this at micropayment prices ($0.005 isn't worth a refund flow) — simplest
   - Verify-then-lock in middleware and commit only after handler succeeds — complex
   - Recommended: log payment on entry, run handler, include tx hash in response, accept rare loss

3. **Bot exemption**: bot players are identified from `SandboxPlayer.is_bot`. Since middleware runs before auth resolution, either:
   - Do a fast DB lookup on `player_id` claim in the request body, or
   - Let bot requests carry a special internal header set by `sandbox_runner.py` and `sandbox_bot.py`, and exempt that header (similar to `ADMIN_API_KEY` exemption)

## Phased Rollout

### Phase 1 — API paywall, off-chain game (implement now)
- Add `X402PaymentMiddleware` behind `X402_ENABLED=false` feature flag
- Gate write actions only; all reads free
- Store payment tx hash in `SandboxTransaction.rentline_payment_id`
- Ship on Base mainnet (lowest fees, widest x402 tooling support)
- CLI surfaces 402 errors gracefully

### Phase 2 — Orbit chain game state (future)
- Game actions become contract calls; x402 per-action gating is no longer appropriate
- Evaluate hosted relayer model: player pays x402 → Rentline submits tx + covers gas
- If relayer is not needed, deprecate x402 and let players call contracts directly
- x402 may still apply to premium off-chain services: analytics API, feed aggregation, AI advisor
